# Архитектура «Тестовый отдел продаж»

Интерактивная песочница ИИ-автоматизации продаж. Посетитель общается с ИИ-менеджером
голосом или текстом и в реальном времени видит, как разговор заносится в CRM,
формируется авто-саммари и считается экономика (ROI).

## Главный принцип

Dasha ведёт разговор. «Магию» (заполнение карточки CRM, саммари, скоринг, SMS, обратный
звонок, ROI) рисует **браузер** — через механизм client-side tools поверх websocket Dasha.
Карточка заполняется не пост-обработкой текста, а в момент, когда ИИ сам вызывает функцию
`update_card`. Бэкенд-хоп для этого не нужен → нулевая задержка на «магию».

## Поток данных

```
┌─────────────────────────── Браузер (SPA) ───────────────────────────┐
│  CRM-макет │ голосовой виджет │ текст │ ROI-панель │ переключатель ниш │
└──────┬───────────────────────────────────────────────────▲───────────┘
       │ 1) POST /api/session/start (ниша, режим)             │ client-side tools
       │    ← короткоживущий Dasha-токен                      │ обновляют стор напрямую
       ▼                                                      │
┌────────────── Бэкенд (Fastify, pm2) ──────────────┐         │
│ минт токена (держит API-ключ) │ лимиты/антиспам    │         │
│ запись лида → Supabase │ ROI-конфиг │ статика      │         │
└───────────────────────────────────────────────────┘         │
       │ 2) wss webCall?token=…                                │
       ▼                                                      │
┌──────────────── Dasha Cloud (РФ) ─────────────────┐         │
│ голос (WebRTC) / текст │ LLM │ TTS                 │         │
│ websocketToolRequest ─────────────────────────────┼─────────┘
│ ← websocketToolResponse                            │
│ conversationResult (авто-саммари) ─────────────────┘
└────────────────────────────────────────────────────┘
```

1. Браузер просит сессию у бэка → бэк проверяет лимиты, минтит короткоживущий токен Dasha.
2. Браузер открывает websocket Dasha напрямую, шлёт `initialize` с `additionalData` (sessionId, ниша).
3. ИИ по ходу диалога шлёт `websocketToolRequest` → браузер исполняет (мутирует стор) → `websocketToolResponse`.
4. В конце приходит `conversationResult` — авто-саммари пишется в карточку.
5. Лид публичного режима браузер отправляет на `POST /api/lead` → Supabase.

## Client-side tools (контракт с агентом Dasha)

| toolName          | args                                   | действие в браузере                    |
|-------------------|----------------------------------------|----------------------------------------|
| `update_card`     | `{ field, value }`                     | заполнить/подсветить поле карточки      |
| `set_summary`     | `{ text }`                             | записать авто-саммари в карточку        |
| `lead_score`      | `{ score, sentiment }`                 | проставить скоринг и тональность        |
| `show_sms`        | `{ text, link }`                       | показать «прилетевшую» SMS              |
| `request_callback`| `{ delaySeconds, reason }`             | запланировать имитацию входящего звонка |
| `transfer`        | `{ reason }`                           | показать перевод на оператора           |

Все toolName и их args задаются в конфиге агента Dasha (по нише) и зеркалятся в
`frontend/src/dasha/tools.ts`.

## Режимы

- **Публичный** — гейт (имя+телефон = первый лид и первая карточка), лимиты по IP,
  кап минут на сессию, лид пишется в Supabase.
- **Презентер** — `?presenter=KEY`, лимиты сняты, можно гонять симулятор.

## Симулятор

Пока нет аккаунта/интеграций Dasha, фронт работает в режиме **simulator**: проигрывает
скриптованный диалог ниши и шлёт те же tool-события в стор. Демо-цикл виден end-to-end
без облака. Реальный Dasha-клиент — drop-in замена (`frontend/src/dasha/DashaClient.ts`).

## Ниши

`стоматология | автосервис | мясной B2B | салон`. Каждая ниша = свой набор полей карточки,
ROI-параметры, сценарий. В Dasha — отдельная Web Integration на нишу
(map `niche → integrationId`) либо одна с `additionalData.niche`.
